# Security

## Overview

familyTraffic's security layer combines system hardening (file permissions, Docker/container checks, network audit), UFW firewall management (default-deny base rules plus per-IP proxy whitelisting), fail2ban brute-force protection (mostly disabled by design in the current architecture), an encryption self-test suite, and informational sudoers guidance. See [[installation]], [[architecture#Network Mode & Traffic Flow]], [[mtproxy#Ports & Firewall]].

## System Hardening

`lib/security_hardening.sh` enforces least-privilege over `/opt/familytraffic` and audits the deployment, invoked during [[installation]] and via `vless security`. It is sourced, not executed directly.

`harden_file_permissions` chmods the install tree to root-owned: dirs `750`, config files `640`, `keys/` to `700` with `600` files, `data/users.json` and `backups/` to `600`, then `chown -R root:root`. `audit_file_permissions` flags non-root ownership, world-readable/writable files, and a keys dir not at `700`. `verify_docker_security` greps `docker-compose.yml` for `cap_drop: ALL`, `no-new-privileges:true`, `read_only: true`, and warns on root user, `network_mode: host`, or `privileged: true` — note the single-container design (see [[docker]]) does run host network. `verify_container_security` inspects running containers for root user, privileged mode, and dropped capabilities. `security_audit` runs all checks (permissions, Docker, container runtime, UFW, network, sensitive data) into a boxed report and returns the issue count; `security_fix` re-applies `harden_file_permissions` + `harden_ufw` then re-audits; `generate_security_report` writes a timestamped `600` report. `audit_sensitive_data` checks for world-readable keys, plaintext passwords in config, and `.env`/`users.json` perms; `audit_network_security` flags exposed `0.0.0.0` ports, host-network containers, and disabled IP forwarding.

## UFW Firewall

UFW is the host firewall layer. Base hardening lives in `security_hardening.sh`; per-IP proxy whitelisting in `lib/ufw_whitelist.sh`. See [[architecture#Network Mode & Traffic Flow]] and [[mtproxy#Ports & Firewall]].

`harden_ufw <vless_port>` sets `default deny incoming` / `default allow outgoing`, allows SSH (`22/tcp`) and the VLESS port (default `443`), applies `ufw limit 22/tcp` rate-limiting, and enables UFW if inactive. `verify_ufw_config` confirms UFW is active, default-deny incoming, SSH allowed and rate-limited, and warns on overly permissive "allow from anywhere" rules. For ACME, `open_port_80_for_acme` temporarily allows `80/tcp` (freeing the port from nginx/apache/Docker if occupied) and `close_port_80_after_acme` removes that rule — see [[certificates]]. `ufw_whitelist.sh` manages access to proxy ports SOCKS5 `1080` and HTTP `8118`: `add_ufw_proxy_rule`/`remove_ufw_proxy_rule` (per-IP, tagged with the comment prefix `VLESS proxy whitelist`), `set_ufw_proxy_ips` (replace whole list), `reset_ufw_proxy_ips`, and `show_ufw_proxy_ips`. `validate_ip` accepts IPv4, `/0`–`/32` CIDR, and basic IPv6. These are exposed through `familytraffic add-ufw-ip|remove-ufw-ip|reset-ufw-ips` (see [[cli-tools]]) and form one layer of a defense-in-depth chain alongside Xray routing, Xray auth, and fail2ban.

## Fail2ban

Two modules generate fail2ban filters and jails, but most jails are intentionally disabled — they cannot ban real attackers in this architecture (see CLAUDE.md and the in-file comments). `lib/fail2ban_setup.sh` covers proxy and Reality jails; `lib/fail2ban_config.sh` covers the reverse-proxy jail.

`fail2ban_setup.sh::setup_fail2ban_for_proxy` installs fail2ban, writes a proxy filter (`familytraffic-proxy`) and jails for SOCKS5 `1080` / HTTP `8118` (plus optional no-TLS `1081`/`8119` when `PROXY_NOTLS_ENABLED=true`), and a Reality filter/jail (`xray-reality`, ports `443`,`8443`). All these jails ship `enabled = false`: nginx terminates TLS and forwards to xray over loopback without the PROXY protocol, so xray only ever sees `127.0.0.1` (banning would break the chain); proxy creds are 128-bit so brute force is infeasible; and HTTP/Reality emit no catchable auth-failure log lines. `verify_fail2ban_jails` only checks that fail2ban is up — the real active protection is the system `sshd` jail. The header comments document how to re-enable (add `proxy_protocol on` + `sockopt.acceptProxyProtocol=true`, point logpath at `access.log`, fix the failregex). Separately, `fail2ban_config.sh::setup_reverseproxy_fail2ban` builds an `enabled = true` jail (`familytraffic-reverseproxy` + a rate-limit variant) protecting nginx reverse-proxy ports (range `9443`–`9452`) by parsing `reverse-proxy-error.log` for HTTP Basic Auth failures and `limit_req`/`limit_conn` violations, with `ufw` ban action (maxretry 5, bantime 3600s, findtime 600s). `add_port_to_jail`/`remove_port_from_jail` dynamically grow/shrink the port list and auto-disable the jail when empty.

## Security Tests

`lib/security_tests.sh` is a standalone `sudo`-run suite (`VLESS Reality VPN - Encryption Security Testing Suite`) validating the encrypted channel end-to-end. It auto-detects production (`/opt/familytraffic`) vs development mode and supports `--quick`, `--skip-pcap`, `--verbose`, `--dev-mode`. See [[testing]] for the BATS unit/integration tests.

It runs eight numbered tests plus an XTLS Vision check: (1) Reality TLS 1.3 config — X25519 key, shortIds, dest, serverNames/SNI; (2) nginx TLS termination (`familytraffic` container running, LE certs present, proxy ports `1080`/`8118` listening — HAProxy was removed in v5.33); (3) traffic encryption via `tcpdump`/`strings`/`tshark` to confirm no plaintext leaks; (4) certificate security — `privkey.pem` must be `600`, chain/SAN/issuer validation; (5) DPI resistance — Reality masquerade dest reachability, SNI match, `nmap` probe, fingerprint; (6) SSL/TLS vulnerabilities — weak ciphers, SSLv2/3, TLS 1.0, PFS on port `8118`; (7) proxy protocol — SOCKS5/HTTP password auth enforced, listen address, UDP disabled, password strength; (8) data-leak detection — config file perms, weak usernames, log scraping, DNS config. `test_xtls_vision_enabled` (TC-01) asserts every Reality client has `flow=xtls-rprx-vision`. Results tally passed/failed/skipped/warnings/critical; exit codes: `0` pass, `1` failure, `2` prerequisites unmet, `3` critical issue. See [[transports]] for XTLS Vision and Reality details.

## Sudoers

`lib/sudoers_info.sh` is purely informational — it never edits `/etc/sudoers`. Sourced by `install.sh`, `display_sudoers_instructions` prints guidance for letting non-root users run the `familytraffic*` CLIs (see [[cli-tools]]).

It explains why management commands need root (Docker, config, key/user management), lists the installed commands, and presents two `/etc/sudoers.d/familytraffic` options: Option 1 passwordless (`%sudo ALL=(ALL) NOPASSWD: ...`, convenient, for trusted single-user VPS) and Option 2 password-required (`%sudo ALL=(ALL) /usr/local/bin/familytraffic*`, more secure, audit-friendly). It walks through applying the file via `visudo` (recommended) or `tee` + `chmod 440`, validating with `visudo -c`, and testing with `sudo -l`. Security notes stress `440`/`400` root-owned perms, trusted-user-only access, wildcard caution, and auditing via `auth.log`. An `offer_automatic_configuration` function exists but is unused by design (manual step only, per Q-002).
