# Proxy Chains

## Overview

familyTraffic can chain client traffic through upstream SOCKS5/HTTP proxies after xray processing (`Client → nginx → xray → external proxy → Internet`). Upstream proxies live in `external_proxy.json`, managed by `familytraffic-external-proxy` and assignable per user. A separate server-level IP whitelist gates who may reach the SOCKS5/HTTP inbounds. HAProxy is deprecated. See [[architecture]], [[security]], [[cli-tools#familytraffic-external-proxy]].

## Per-User Upstream Chains

Upstream proxies are stored in `/opt/familytraffic/config/external_proxy.json` (mode 600), managed by `lib/external_proxy_manager.sh`. Each proxy has a generated id (`proxy-<address>-<6 digits>`), a type, optional auth, TLS, and retry block. Supported types: `socks5`, `socks5s`, `http`, `https`; `socks5s`/`https` set `tls.enabled`.

Proxies are added via `add_external_proxy` (validated by `validate_proxy_config`: type, address ≤253 chars, port 1–65535, auth requiring both username and password). The DB also holds a top-level `enabled` flag and a `routing` block (`mode: all-traffic`, `fallback: retry-then-block`). `set_active_proxy` marks one proxy `active` and clears the rest; `test_proxy_connectivity` runs three curl attempts against `https://1.1.1.1` and records latency/status in metadata. `generate_xray_outbound_json` emits the xray outbound (tag `external-proxy`, protocol `socks` or `http`, with `tls`/`allowInsecure` stream settings when applicable).

Per-user assignment (v5.24) is tracked by `external_proxy_id` on each entry in `data/users.json`: empty/null means direct routing, otherwise the user's traffic chains through that proxy. `get_routing_status` summarizes direct vs proxied users and groups users by proxy. See [[user-management]].

## CLI Workflow

`familytraffic-external-proxy` (in `scripts/`) is the root-only CLI front end; it sources `external_proxy_manager.sh` and `xray_routing_manager.sh`. Commands: `add` (6-step interactive wizard), `list`, `show <id>`, `switch <id>`, `update <id>`, `remove <id>`, `test <id>`, `enable`, `disable`, `status`.

`add` walks through type, address/port, TLS SNI + allow-insecure, and optional auth, then calls `add_external_proxy`, tests it, and offers to activate and enable routing. `switch` runs `set_active_proxy` + `update_xray_outbounds` to rewrite the xray outbound. `enable`/`disable` call `enable_proxy_routing`/`disable_proxy_routing` and then automatically `docker exec familytraffic supervisorctl restart xray`, waiting for the container to report `running`. Other mutating commands print a manual restart reminder. See [[cli-tools#familytraffic-external-proxy]], [[services]].

## Proxy Whitelist

The proxy whitelist (`lib/proxy_whitelist.sh`) is a server-level allowlist of source IPs permitted to reach the SOCKS5 (1080) and HTTP (8118) proxy inbounds — it is global, not per-user (per-user whitelisting from v3.5 was dropped because xray's `user` field only works for VLESS, not HTTP/SOCKS5).

IPs live in `/opt/familytraffic/config/proxy_allowed_ips.json` (mode 600), defaulting to `127.0.0.1`. `validate_ip` accepts IPv4/IPv6 with optional CIDR. Functions `set`, `add`, `remove`, `reset`, `show`, and `regenerate` mutate the list under an `flock` lock; `remove` refuses to delete the last IP. After any change, `regenerate_proxy_routing` rewrites the xray `routing` section with a single `field` rule (`inboundTag: [socks5-proxy, http-proxy]`, `source: <ips>`, `outboundTag: direct`) and `reload_xray` signals SIGHUP (fallback restart) to the xray process. The script runs standalone or sourced; CLI subcommands are `init|show|set|add|remove|reset|regenerate`. See [[security]].

## HAProxy Config (Deprecated)

`lib/haproxy_config_manager.sh` is deprecated as of v5.33. HAProxy was removed when the stack collapsed into the single `familytraffic` container; SNI routing and TLS termination are now handled by nginx, generated by `lib/nginx_stream_generator.sh`.

All HAProxy functions (`generate_haproxy_config`, `add_reverse_proxy_route`, `remove_reverse_proxy_route`, `validate_haproxy_config`, `reload_haproxy`, `check_haproxy_status`, `list_haproxy_routes`) are retained only as no-op stubs for call-site compatibility — each logs a deprecation warning and returns 0. References to a "Client → HAProxy → xray" chain in older module headers are historical; the live flow is `Client → nginx → xray → external proxy`. See [[architecture]], [[transports]].
